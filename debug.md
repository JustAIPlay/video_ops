优化建议 (可以做得更好的地方)
为了让这个功能在落地时更稳健、更有用，我有以下建议：
1. 业务逻辑与数据层面
缺少“上下文关联”的输入定义:
问题: 增长黑客 (Growth Hacker) 要提出“前3秒加入实物展示”这样的假设，它需要知道视频的内容是什么。
建议: 在 System Prompt 中，除了传入数据（播放量），还需要传入 视频标题、标签、甚至 OCR 提取的关键帧描述（如果有）。如果目前只有数据，增长黑客的建议可能会变得“万金油”（如：“建议优化标题”这种正确的废话）。
修正: 明确 Agent 的输入 Context 包含哪些字段。
行动项 (Action Items) 的可执行性:
问题: 总结卡片里的“明日行动计划”目前是文本。
建议: 将 ActionItem 数据结构化，允许用户点击“一键添加到排期”。例如，策略师建议“明天 19:30 发布视频 B”，用户点击确认后，直接修改 Phase 1 的排期表。这将打通复盘与执行的闭环。
历史记忆 (Memory):
建议: Agent 需要知道昨天的复盘结果。比如昨天增长黑客建议了“A/B测试标题”，今天的分析师应该主动汇报“昨天测试的标题效果如下...”。这需要一个简单的向量库或 JSON 历史记录存储。
2. 交互体验层面 (Frontend)
打断与插话机制:
场景: 会议进行中（例如数据分析师正在说话），用户突然想问一个问题。
建议: 明确 UI 上是“禁用提问直到说完”还是“允许打断”。建议 MVP 阶段采用 “轮流发言制”，Agent 说完后，用户才可以提问，避免 SSE 流中断带来的技术复杂性。
移动端适配:
注意: 现在的设计是三栏布局或宽卡片（max-w-[1600px]）。在移动端，Agent 头像横排可能会挤，建议在移动端改为折叠式或垂直堆叠布局。
3. 技术实现层面 (Backend)
并发 vs 串行:
问题: 如果三个 Agent 串行生成，耗时可能较长（每人 30 秒，开会就要 2 分钟）。
建议:
数据分析和增长黑客可以并行思考（预加载），但串行输出（为了阅读体验）。
或者在 LoadingScreen 阶段，后端就已经开始并发请求 LLM 生成所有内容，前端只是负责按顺序“播放”出来。这能极大提升体感速度。
错误处理:
场景: 如果飞书接口挂了，或者 LLM 超时。
建议: UI 需要由 Retry 机制，或者 Agent 能输出“我暂时无法获取数据，请稍后再试”。
